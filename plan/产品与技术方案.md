## 一、产品定位

### 1.1 一句话定位
**一个知识型个人博客，文章即知识库，任何访客都能向 AI 提问，AI 基于博主的文章回答。**

### 1.2 核心理念
> 传统的个人博客是"静"的——访客只能读。这个博客是"活"的——访客读完还能问，AI 替博主回答。

- **内容即知识库**：博主发布的每一篇文章自动向量化入库，无需额外维护知识库。
- **访客可问**：完全公开，无需登录即可向 AI 提问，AI 基于文章内容作答并引用出处。
- **频道化场景**：博主的不同内容板块（个人经验、深圳政策研究等）成为"频道"，每个频道有独立的 AI 问答入口与预设人设。HR 助手、政策查询自然融入为频道。
- **语音通话**：问答支持豆包式实时语音，访客可解放双手与博主 AI 对话。

### 1.3 与 v1.0 的关系

| 维度 | v1.0 | v2.0（本次） |
|------|------|------|
| 主站形态 | 工作台门户 | 个人博客 |
| 知识库来源 | 手动上传文档 | 博客文章自动入库 |
| 问答对象 | 内部用户（HR/法务） | 全网公开访客 |
| 模块关系 | 两个独立 ToB 模块 | 频道化融入博客 |
| HR 助手 | 独立模块 | "个人经验"频道 + 代答人设 |
| 政策查询 | 独立模块 | "深圳政策"频道 + 采集管道 |
| 语音通话 | 模块一专属 | 全站问答的语音模式 |

## 二、核心场景

### 场景一：访客读文章 → 深入提问
> 访客读到一篇关于"候选人评估方法"的文章，文末有"问 AI"入口，提问"那如果是初级岗位怎么评估？"，AI 基于这篇及关联文章回答。

### 场景二：访客全站搜索式问答
> 访客不读文章，直接在首页问答框提问"博主对远程办公的看法？"，AI 检索全站文章综合回答，附带引用文章链接。

### 场景三：HR 通过"个人经验"频道向"虚拟你"提问（原 HR 助手）
> HR 访客进入"个人经验"频道，AI 以博主人设代答（"我"口吻），解决招聘协作中的问题。

### 场景四：政策查询通过"深圳政策"频道（原政策查询）
> 访客进入"深圳政策"频道，查询人才补贴/社保/入户等政策，AI 基于博主整理+采集的政策库回答，标注文号出处。

### 场景五：语音通话
> 访客点击"语音对话"，像用豆包一样与博主 AI 语音交流，适合移动端或解放双手场景。

### 场景六：博主内容经营
> 博主在后台写文章、发布，系统自动入库；查看问答数据看板，了解访客最关心什么，反哺内容选题。

## 三、信息架构

### 3.1 三层结构

```
博客主站（公开阅读）
├── 文章（自动入库为知识库）
├── 频道（内容分组 + 独立问答）
│   ├── 个人经验（= 原 HR 助手，代答人设）
│   ├── 深圳政策（= 原政策查询，采集+录入）
│   └── 可扩展更多频道
└── AI 问答（贯穿全站）
    ├── 范围：全站 / 某频道 / 单篇文章
    └── 模式：文字 / 语音通话
```

### 3.2 频道机制（统一原两个模块）

频道是博客的内容分组，每个频道可配置：
- **名称与简介**：如"个人经验""深圳政策"
- **包含文章**：按标签或手动归入
- **独立知识库**：频道内文章 + 频道专属数据（如政策采集）
- **AI 人设**：频道问答的人设（如"个人经验"频道以博主第一人称代答）
- **问答入口**：频道页有独立的问答区

| 频道 | 对应原模块 | 知识库来源 | 人设 |
|------|-----------|-----------|------|
| 个人经验 | HR 助手 | 博主个人经验文章 | 以"我"代答 |
| 深圳政策 | 政策查询 | 政策文章 + 官网采集 + API + 人工录入 | 客观解读，标注文号 |
| （可扩展） | - | - | - |

### 3.3 问答范围模型

AI 问答支持三种检索范围，由入口决定：

| 入口 | 范围 | 场景 |
|------|------|------|
| 全站问答框 | 全站所有已发布文章 | 综合提问 |
| 频道问答区 | 该频道内文章+专属数据 | 场景化提问 |
| 文章末尾"问AI" | 单篇文章 + 关联推荐 | 深入提问 |

## 四、功能设计

### 4.1 博客主站（公开）

| 功能 | 说明 |
|------|------|
| 文章列表 | 按时间/分类/标签浏览，支持搜索 |
| 文章详情 | Markdown 渲染、代码高亮、目录、阅读时长 |
| 分类与标签 | 内容组织，标签同时用于频道归集 |
| 文章末尾问答 | 每篇文章底部"问 AI"入口，基于该文章问答 |
| 关于页 | 博主介绍、联系方式 |
| 全站问答入口 | 首页/导航的醒目问答入口 |

### 4.2 AI 问答（公开）

| 功能 | 说明 |
|------|------|
| 文字问答 | 流式输出、多轮上下文、引用溯源 |
| 语音通话 | 豆包式实时语音、可打断、双向字幕 |
| 范围切换 | 全站 / 频道 / 文章三级范围 |
| 引用溯源 | 答案附带文章链接，可跳转原文 |
| 主动追问 | 信息不足时 AI 反问澄清 |
| 推荐提问 | 根据当前文章/频道推荐问题 |
| 会话历史 | 本地存储（无需登录），登录后云端同步 |

### 4.3 频道（场景化）

| 功能 | 说明 |
|------|------|
| 频道主页 | 频道简介 + 文章列表 + 独立问答入口 |
| 频道人设 | 配置该频道 AI 回答风格与口吻 |
| 频道知识库 | 文章 + 专属数据（如政策采集） |
| 政策频道专属 | 多维检索、时效标记、数据采集管道（同 v1.0） |

### 4.4 博主后台（私有）

| 功能 | 说明 |
|------|------|
| 文章管理 | 写作（Markdown/富文本）、草稿、发布、分类标签 |
| 知识库管理 | 查看入库状态、重建向量、手动上传补充文档 |
| 频道管理 | 创建频道、归集文章、配置人设 |
| 人设配置 | 全站与各频道人设（口吻、口头禅、禁忌） |
| 数据看板 | 访问量、热门文章、热门问题、问答统计 |
| 政策采集 | 政策频道的爬虫/API/录入管理（同 v1.0） |

### 4.5 权限模型

| 角色 | 能力 |
|------|------|
| 访客（未登录） | 读文章、公开问答（文字+语音）、看历史（本地） |
| 注册访客 | 同上 + 云端会话历史 |
| 博主 | 全部 + 后台管理 |

## 六、技术选型调整

v2.0 从"内部工作台"变为"公开博客 + AI 问答"，前端架构、Markdown 渲染、编辑器、搜索均有调整。后端与 AI 链路主体不变。

### 6.1 复用（不变）

| 层 | 选型 | 说明 |
|----|------|------|
| 后端 | Python + FastAPI | 不变 |
| RAG | Milvus + BGE-M3 + bge-reranker | 不变 |
| LLM | 国产 API（通义/DeepSeek） | 不变 |
| 语音 | 流式 ASR/TTS + VAD 可打断 | 不变 |
| 部署 | 本地先行 Docker Compose，后上云 | 不变 |

### 6.2 前端架构调整（重要）

| 项 | v1.0 | v2.0 | 原因 |
|----|------|------|------|
| 框架 | Vite + React SPA | **Next.js 14 (App Router)** | 博客需 SSR/SSG 做 SEO，本期就做 |
| 路由 | React Router v6 | Next.js App Router | 文件式路由，SSR/SSG 原生支持 |
| 渲染策略 | 纯 CSR | **SSG 博客页 + CSR 问答页** | 文章页静态生成（SEO+快），问答页客户端渲染（流式） |
| 状态管理 | Zustand | Zustand（不变） | 问答交互态仍需要 |
| UI 组件 | AntD 5 | AntD 5（不变） | 后台沿用；博客前台可配 CSS Modules 做定制风格 |
| 数据获取 | Axios + TanStack Query | **Next.js fetch + RSC + TanStack Query** | 服务端组件直接取数，客户端交互用 Query |

**渲染策略细分**：
- 博客文章页 `/posts/:slug`：**SSG**，构建时/按需生成静态 HTML，SEO 友好、首屏零白屏。
- 频道页、首页：SSG 或 ISR（增量静态再生），内容更新时自动重新生成。
- 问答页 `/ask`、语音页、后台：**CSR**，客户端渲染，支持流式与交互。
- 博主后台：CSR，登录态私有，不需 SEO。

### 6.3 Markdown 渲染调整（重要）

博客文章渲染与对话答案渲染是两套需求，分开选型：

#### 博客文章展示（SSG 预渲染）
| 能力 | v1.0 | v2.0 | 原因 |
|------|------|------|------|
| 渲染引擎 | react-markdown (CSR) | **MDX + next-mdx-remote** | 支持 SSG 预渲染，可在 Markdown 中嵌入 React 组件 |
| 代码高亮 | rehype-highlight (highlight.js) | **Shiki** | VS Code 同款，主题质感好，SSR 友好，博客质感拉满 |
| 标题锚点/目录 | 无 | rehype-slug + rehype-autolink-headings | 博客需要 TOC 目录与锚点跳转 |
| 图片 | 无 | next/image + react-medium-image-zoom | 懒加载 + 点击放大 |
| Mermaid 图表 | 无 | rehype-mermaid | 技术博客常用 |
| 数学公式 | 无 | remark-math + rehype-katex | 技术写作需要 |
| 阅读时长 | 无 | remark-reading-time | 博客标配 |

#### 对话答案渲染（CSR 流式）
| 能力 | 选型 | 原因 |
|------|------|------|
| 渲染引擎 | react-markdown + remark-gfm | 流式增量渲染，保留不变 |
| 代码高亮 | rehype-highlight (highlight.js) | 轻量，流式场景够用（Shiki 流式渲染较重） |
| 策略 | 段落缓冲 | 避免表格/代码块流式时闪烁 |

> 博客文章用 Shiki（静态、质感优先），对话答案用 highlight.js（轻量、流式优先），各取所长。

### 6.4 编辑器选型（新增）

博主写作采用**源码 + 预览分屏**模式：

| 项 | 选型 | 原因 |
|----|------|------|
| 编辑器 | **Bytemd**（字节开源） | 专为博客设计，源码+预览分屏，插件生态好（GFM/数学/Mermaid/高亮） |
| 备选 | CodeMirror 6 + 自建预览 | 更灵活但开发量大 |
| 图片上传 | 复用后端对象存储 | 粘贴/拖拽上传，返回 URL 插入 Markdown |
| 草稿自动保存 | 防抖 + 后端草稿接口 | 30s 自动保存 |

> 不用 Tiptap（WYSIWYG）。技术博主习惯 Markdown 源码，Bytemd 更契合，且与展示端 MDX 渲染一致。

### 6.5 其他新增

| 能力 | 选型 | 说明 |
|------|------|------|
| 博客 CMS | FastAPI + 文章 CRUD | 草稿/发布/分类/标签 |
| 公开访问治理 | Redis 限流 + 验证码 | 防问答被滥用、防爬虫 |
| 文章自动入库 | Celery 任务 + 文档解析 | 发布文章时触发向量化 |
| 全文搜索 | **Meilisearch** | 开箱支持中文分词、即时搜索；PG 全文索引中文支持弱，弃用 |
| SEO 增强 | next-sitemap + 元数据 API | 自动 sitemap、og 标签、结构化数据 |

| 项 | v1.0 | v2.0 |

## 七、关键设计决策

### 7.1 为什么用"频道"统一原两个模块
- HR 助手与政策查询本质都是"某类内容 + AI 问答"，差异仅在数据源与人设。
- 频道抽象后，新增场景（如"读书笔记""技术专栏"）只需建频道，无需开发新模块。
- HR 助手 = "个人经验"频道 + 代答人设；政策查询 = "深圳政策"频道 + 采集管道。

### 7.2 为什么文章自动入库而非手动上传
- 博主只关心写文章，不关心"维护知识库"。
- 发布即入库，零额外成本，保证知识库与博客内容同步。
- 手动上传保留为补充（如导入历史笔记、政策 PDF），支持 MD/TXT/PDF/DOCX 多格式批量上传，源文件归档 MinIO 便于重解析。

### 7.3 为什么公开问答
- 知识型个人站的核心价值是"让博主的知识可被对话式获取"。
- 公开问答降低门槛，扩大影响力，符合个人品牌建设目标。
- 通过限流 + 引用溯源 + 拒答兜底控制风险。

### 7.4 语音通话为何升为全站能力
- 原本仅 HR 助手可用，但语音是"与博主 AI 对话"的自然方式，不应局限于一个频道。
- 全站问答与频道问答都支持切换语音模式，体验一致。

## 九、风险与对策

| 风险 | 对策 |
|------|------|
| 公开问答被滥用/爬虫 | Redis 限流（IP/会话）+ 验证码 + 异常检测 |
| LLM 成本失控 | 缓存常见问题 + 限流 + 降级（超限改文字摘要） |
| 文章入库延迟 | 异步队列 + 发布后秒级入库（小文档） |
| 公开内容合规 | 博主审核 + 敏感词过滤 + 拒答兜底 |
| 语音成本 | 语音模式限流更严 + 降级文字 |
