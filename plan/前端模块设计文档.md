## 一、概述

### 1.1 文档目的
定义"个人博客 + AI 知识库问答"前端的整体架构、目录结构、路由、状态管理、核心组件、通信层与构建部署。

### 1.2 前端职责范围
- 公开博客站点：文章浏览、频道、关于、搜索
- AI 问答（贯穿全站）：文字流式问答 + 豆包式语音通话，范围三级（全站/频道/文章）
- 博主后台：文章管理、知识库、频道、人设、数据看板、政策采集

### 1.3 设计原则
1. **SSG/CSR 分层**：博客内容页静态化（SEO），问答交互页客户端渲染（流式）。
2. **内容即知识库**：文章发布自动入库，前端无需维护独立知识库上传流（保留补充上传）。
3. **流式优先**：文字与语音均为流式交互，组件层统一抽象。
4. **频道化**：原 HR 助手、政策查询统一为频道，前端复用同一套频道模板。
5. **本地/云兼容**：配置外置，构建产物不绑定部署环境。

- `app/` 只做路由与页面组装，逻辑下沉到 `lib/` 与 `components/`。

- 服务端组件（RSC）直接 `await fetch`，客户端组件用 `'use client'` 声明。

## 四、路由与渲染策略

### 4.1 路由表与渲染方式

| 路由 | 渲染 | 鉴权 | 说明 |
|------|------|------|------|
| `/` | SSG/ISR | 公开 | 首页,Hero + AskBox + 最新 3 篇 Feature 卡片 + 历史存档列表;底部"浏览全部文章"指向首个频道 |
| `/posts/[slug]` | **SSG** | 公开 | 文章详情,`generateStaticParams` 预生成;含 TOC、上一篇/下一篇、文末 AskBox |
| `/channel` | ISR | 公开 | 频道单页,query param 驱动：`?channel=xxx` 切换频道、`&tag=yyy` 标签筛选;横向 chips 不跳页,切换时仅更新 query param 触发 RSC 重渲染 |
| `/channel/[slug]` | — | 公开 | 旧路由,`redirect` 到 `/channel?channel=slug&tag=...`,保留 tag 参数兼容历史链接 |
| `/ask` | **CSR** | 公开 | 问答页（流式交互） |
| `/ask/voice` | **CSR** | 公开 | 语音通话页 |
| `/search` | CSR | 公开 | Meilisearch 搜索(P3 暂缓,目前为 placeholder) |
| `/about` | SSG | 公开 | 关于 |
| `/admin/*` | CSR | 博主 | 后台，登录鉴权 |
| `/login` | CSR | 公开 | 登录 |
| `/sitemap.xml` | SSG | 公开 | Next.js Metadata API 自动生成 |
| `/robots.txt` | SSG | 公开 | Next.js Metadata API 自动生成 |

> **路由精简说明(v2.1)**:移除了独立的 `/posts` 文章列表页与 `/tags`、`/tags/[tag]` 标签云/筛选页。文章浏览与筛选统一收敛到频道页,通过 URL `?tag=xxx` 进行标签筛选,通过顶部 chips 进行频道切换。"文章即知识库"定位下,频道是唯一文章组织维度,跨频道的全站列表不再需要。
>
> **频道单页化(v2.2)**:多个频道页面合并为单个 `/channel`,频道切换通过横向 chips + query param 完成,**不进行页面跳转**,仅触发 RSC 重渲染,布局保持稳定。导航栏不再逐个列出频道,改为单个"频道"入口指向 `/channel`。旧路由 `/channel/[slug]` 保留为重定向以兼容历史链接（sitemap、文章页"返回频道"、首页"浏览全部"等已同步更新为 query param 格式,避免重定向跳转）。

### 4.2 SSG 实现要点
- 文章详情页用 `generateStaticParams` 预生成所有已发布文章的静态页。
- 文章更新时通过 **按需重新生成**（On-Demand ISR）：后端发布文章后调 Next.js revalidate API 触发重新生成。
- 首页/频道用 ISR，设定刷新间隔或按需触发。
- 缓存采用 tag-based(`tags: ['posts', 'post:{slug}']`),revalidate 时按 tag 精准失效。

### 4.3 SEO 要点
- 每页用 Next.js Metadata API 配 `title`/`description`/`openGraph`。
- 文章详情 `generateMetadata` 设置 `openGraph.type='article'`、`publishedTime`、`tags`。
- `app/sitemap.ts` + `app/robots.ts` 由 Next.js Metadata API 自动生成 sitemap.xml 与 robots.txt(静态页 + 动态文章/频道)。
- 文章页加 JSON-LD 结构化数据（Article schema）。
- MDX 内容 SSG 预渲染为 HTML，搜索引擎可直接抓取。

### 4.4 鉴权
- 公开站点无鉴权。
- `/admin/*` 用中间件（middleware.ts）拦截，未登录跳 `/login?redirect=...`。
- 博主 token 存 httpOnly cookie，中间件校验。

## 五、状态管理

### 5.1 状态划分原则
- **服务端状态**（文章列表、频道、配置）→ RSC 直取或 TanStack Query。
- **客户端交互状态**（当前会话、输入草稿、通话状态）→ Zustand。
- **局部 UI 状态** → 组件内 useState。

### 5.2 Zustand Store 设计

#### chatStore（客户端组件）
```ts
interface ChatState {
  scope: { type: 'global' | 'channel' | 'article'; refId?: string };
  sessionId: string | null;
  messages: ChatMessage[];
  streaming: boolean;
  followUps: string[];
  setScope: (s: ChatState['scope']) => void;
  appendMessage: (m: ChatMessage) => void;
  appendToken: (id: string, token: string) => void;
}
```

#### voiceStore（客户端组件）
```ts
interface VoiceState {
  status: 'idle' | 'connecting' | 'connected' | 'speaking' | 'listening' | 'ended' | 'error';
  micMuted: boolean;
  transcript: string;
  aiText: string;
  interruptible: boolean;
  setStatus: (s: VoiceState['status']) => void;
  appendTranscript: (t: string) => void;
  appendAiText: (t: string) => void;
  reset: () => void;
}
```

> Zustand 仅在客户端组件中使用，服务端组件不依赖。

## 六、核心组件设计

### 6.1 博客文章渲染（SSG）

#### ArticleContent（MDX 容器）
```
ArticleContent
├── TableOfContents          # 目录（锚点跳转，滚动高亮）
├── MDXRemote                # next-mdx-remote 渲染
│   └── 自定义组件映射：<Code>, <Callout>, <Image>, <Mermaid>
├── ReadingTime              # 阅读时长
└── ArticleAskEntry          # 文末"问 AI"入口
```

- 文章 Markdown 在服务端通过 `serialize`（next-mdx-remote）预渲染为 HTML，Shiki 高亮代码。
- 客户端只做目录滚动监听、图片缩放等交互。
- MDX 支持自定义组件（如 `<Callout>` 提示框），提升博客表达力。

#### 渲染管线
```
Markdown 源码 → remark/rehype 插件链 → Shiki 高亮 → MDX 序列化 → SSG HTML
插件：remark-gfm, remark-math, rehype-katex, rehype-slug,
      rehype-autolink-headings, rehype-mermaid, remark-reading-time
```

### 6.2 对话组件体系（文字问答，CSR）

```
ChatContainer                    # 'use client'
├── MessageList
│   └── MessageItem
│       ├── StreamingText        # react-markdown 流式渲染
│       ├── Citation             # 引用溯源（指向文章链接）
│       └── FollowUpQuestions    # 推荐追问
├── InputBox
└── VoiceEntryButton             # 切语音
```

#### StreamingText（流式渲染）
- 接收 token 流，逐字追加渲染 `react-markdown`。
- 用 `requestAnimationFrame` 节流，按段落缓冲避免表格/代码块闪烁。
- 末尾光标 `▋` 闪烁表示生成中。
- 代码高亮用 rehype-highlight（轻量，适合流式）。

#### Citation（引用溯源）
- 每条 AI 消息携带 `citations: { articleId, title, slug, snippet }[]`。
- 点击展开原文片段，"查看原文"跳转 `/posts/:slug`。

### 6.3 提示询问·主动追问

```ts
type ChatMessage =
```

- `clarify` 消息渲染为 `ClarifyCard` 带选项卡片，点选项自动回复。

### 6.5.1 文件上传组件（后台）

#### 文章页 MD 批量上传
```
PostUploadModal                  # 'use client'
├── 频道选择 Select              # 必填，决定草稿归属频道
├── Upload.Dragger               # multiple, maxCount=20, accept=".md,.markdown"
├── 文件列表                     # 显示文件名、大小、校验状态
├── 上传按钮                     # 全部校验通过后可点
└── 结果展示                     # 成功列表（可跳转编辑）+ 失败列表（含原因）
```

- 限制：仅 `.md` / `.markdown`，单文件 ≤ 5MB，UTF-8 编码
- 接口：`POST /api/admin/posts/upload`，FormData 字段 `files` + `channel_id`
- 返回：`{created: ArticleAdmin[], failed: [{filename, reason}]}`
- 成功后每篇跳转 `/admin/posts/{id}/edit`，失败项显示原因供用户修正后重试

#### 知识库多格式多文件上传
```

├── 频道选择 Select              # 必填，决定知识库归属频道
├── Upload.Dragger               # multiple, maxCount=20, accept=".md,.txt,.pdf,.docx"
├── 文件列表                     # 显示文件名、格式图标、大小、校验状态
├── 上传按钮                     # 全部校验通过后可点
└── 结果展示                     # 成功列表（含入库状态）+ 失败列表（含原因）
```

- 格式与限制：
  - `.md` / `.txt`：≤ 5MB
  - `.pdf` / `.docx`：≤ 20MB
- 接口：`POST /api/admin/knowledge/upload`，FormData 字段 `files` + `channel_id` + 可选 `title`
- 返回：`{created: KnowledgeDoc[], failed: [{filename, reason}]}`
- 上传是同步入库（含解析→分块→embedding→Milvus），timeout 120s
- 成功后列表展示入库状态（indexed/partial/failed），partial/failed 可点"重建"按钮

### 6.6 频道组件

```
ChannelPage                      # 频道单页(ISR,文章浏览与筛选的唯一入口,query param 驱动)
├── ChannelHeader                # 频道名/简介/人设提示/AskBox(scope=channel),随当前频道变化
├── FilterBar                    # 粘性筛选栏(sticky top-16)

```

- 频道是"文章即知识库"定位下唯一的文章组织维度;不再有独立的 `/posts` 全站列表或 `/tags` 标签云。

- 默认频道取列表第一个;无频道数据时展示空状态。
- 频道问答复用 `ChatContainer`，仅 `scope` 不同。
- 政策频道额外提供多维检索入口（时效/主题/单位筛选,P1)。

## 八、语音通话前端实现（豆包式体验）

### 8.1 交互状态机

```
        ┌─────────┐  点击通话   ┌────────────┐
        │  idle   │ ──────────► │ connecting │
        └─────────┘             └─────┬──────┘
            ▲                         │ WS 建立 + 麦克风授权
            │ 挂断                     ▼
        ┌───┴──────┐  用户开口   ┌──────────┐  AI 开口   ┌──────────┐
        │  ended   │ ◄────────── │ listening │ ─────────► │ speaking │
        └──────────┘             └──────────┘             └────┬─────┘
              ▲                                                │
              └─────────── 用户开口打断 ──────────────────────┘
```

### 8.2 音频采集
```ts
async function getMicStream(): Promise<MediaStream> {
  return navigator.mediaDevices.getUserMedia({
    audio: {
      channelCount: 1,
      sampleRate: 16000,
      echoCancellation: true,   // 回声消除（关键，否则 AI 声音被回采）
      noiseSuppression: true,
      autoGainControl: true,
    },
  });
}
```
- 用 `AudioWorklet` 采集 PCM，16kHz 单声道，每帧 ~20ms。
- **回声消除必开**：否则 TTS 声音被回采触发误打断。

### 8.3 前端 VAD
- `silero-vad`（WASM）判断用户说话起止。
- 用户开口 → 触发 `interrupt` → 后端停 TTS。
- 用户停顿超 600ms → 触发 `utterance_end` → 后端生成回复。

### 8.4 WebSocket 协议（二进制 + 控制帧）

| 首字节 | 类型 | 方向 | 内容 |
|--------|------|------|------|
| `0x01` | 音频帧 PCM | C→S | 16kHz PCM |
| `0x02` | ASR 文本增量 | S→C | `{ text }` |
| `0x03` | LLM token | S→C | `{ token }` |
| `0x04` | TTS 音频帧 | S→C | AI 语音 PCM/Opus |
| `0x10` | 控制帧 JSON | 双向 | 见下 |

控制帧：
```ts
// C→S
{ type: 'start_call' }
{ type: 'interrupt' }
{ type: 'mute', muted: boolean }
{ type: 'end_call' }

// S→C
{ type: 'asr_final', text }
{ type: 'ai_start' }
{ type: 'ai_end' }
{ type: 'status', status: 'listening'|'speaking' }
{ type: 'error', message }
```

### 8.5 TTS 播放队列
- 收到 `0x04` → 入 `AudioWorklet` 播放队列，边收边播，首音到达即播。
- 收到 `interrupt` → 清空队列停止播放。

### 8.6 useVoiceCall Hook
```ts
function useVoiceCall() {
  return {
    status, start, hangup, toggleMute, interrupt,
    micLevel, aiLevel,        // 波形电平
    transcript, aiText,       // 双向字幕
    fallbackToText,
  };
}
```

### 8.7 降级与兜底
- 麦克风授权失败 → 提示回退文字。
- WS 断连 → 重连 1 次，失败回退文字。
- 不支持 AudioWorklet → 降级 ScriptProcessor。

### 8.8 性能要点
- 音频处理在 Worklet 线程，主线程只做 UI。
- 波形用 Canvas + RAF，不触发 React 重渲染。
- 转写文本用 ref + DOM 直更，避免每 token 重渲染。

## 十二、开发规范

1. **服务端/客户端组件**：默认 RSC，需要交互才加 `'use client'`，避免不必要的水合。
2. **命名**：组件 PascalCase，hooks 以 `use` 开头。
3. **类型**：禁止 `any`，API 响应有 interface。
4. **提交**：Conventional Commits，Husky + lint-staged 校验。
5. **样式**：博客前台 CSS Modules 隔离风格，后台 Tailwind。
6. **流式渲染**：统一走 `useStreamRender`，禁止组件直接操作 WS。
7. **音频**：逻辑封装在 `utils/audio.ts` 与 `lib/services/voiceCall.ts`，组件不直接接触 `AudioContext`。
8. **MDX**：博客文章插件链与 Bytemd 编辑器插件链保持一致，保证所见即所得。
