## 一、设计目标

后端要支撑"**文章即知识库，访客可问，AI 代答**"的产品形态，具体承担：

1. **博客 CMS**：文章 CRUD、草稿/发布、分类标签、频道管理。
2. **自动入库**：文章发布即触发解析 → 分块 → 向量化 → 写入 Milvus，无需人工维护知识库。
3. **RAG 问答**：三级范围检索（全站 / 频道 / 文章）、引用溯源、追问、主动澄清。
4. **语音通话**：流式 ASR / LLM / TTS 闭环，VAD 可打断。
5. **政策采集**：政策频道专属的爬虫 / API / 人工录入三源管道。
6. **公开访问治理**：限流、会话、合规兜底。
7. **博主后台**：知识库管理、人设配置、数据看板、站点设置。

### 设计原则

- **博客优先**：CMS 与公开 API 是地基，RAG 是叠加能力；RAG 失败不能让博客宕机。
- **范围路由统一**：全站 / 频道 / 文章共用一套 RAG 管道，仅向量集合 partition 不同。
- **流式优先**：文字与语音均为流式，接口风格统一为"事件流"。
- **可降级**：LLM 超限 / 语音失败 → 自动降级到文字摘要。
- **本地先行**：本地 Docker Compose 可全栈跑通，云端为后续阶段。

### 模块边界

| 模块 | 职责 | 对外接口 |
|------|------|---------|
| `api_gateway` | 路由聚合、鉴权、限流、审计 | HTTP |
| `cms` | 文章/频道/人设/站点配置 CRUD | REST |
| `knowledge` | 文档解析、分块、向量化、重建 | Celery + REST 触发 |
| `rag` | 检索 + 重排 + LLM 编排、引用溯源 | WS（文字）/ 内部调用（语音） |
| `voice` | ASR / TTS / VAD / 打断 | WS（语音） |
| `policy_collector` | 政策频道数据采集 | Celery + REST 触发 |
| `search` | Meilisearch 索引同步与查询 | REST |
| `common` | 鉴权、限流、日志、配置、错误 | 横切 |

### 进程形态

- **Web 进程**（uvicorn）：FastAPI HTTP + WebSocket，单进程多 worker。
- **Worker 进程**（celery worker）：入库、采集、批量重排、定时任务。
- **调度进程**（celery beat）：定时采集、过期会话清理、向量重建。
- **流式进程**（uvicorn，独立端口或同进程 WS 路由）：长连接，单独水平扩展。

> 本地用 Docker Compose 单机跑全栈；云端把 Web/Worker/流式拆为独立 Deployment。

### 4.3 Milvus 集合设计

单个 collection `inkgrid_chunks`，用 partition 表达三级范围：

| Partition | 内容 | 路由场景 |
|-----------|------|---------|
| `global` | 所有已发布文章分块 | 全站问答 |
| `channel:{slug}` | 该频道所有文章 + 频道专属数据（如政策条目） | 频道问答 |
| `article:{slug}` | 单篇文章分块（与 global 重叠，便于精准路由） | 文末问 AI |

> 三级范围路由实际为查询时指定 `partition_names`，避免多集合维护。`article` partition 可选——单文章问答可直接 filter `article_slug == x`。

**Schema：**

```python
# Milvus 字段
id           VARCHAR(64) pk        # 与 chunks.embedding_id 对齐
doc_id       VARCHAR(64)
channel      VARCHAR(32)
article_slug VARCHAR(64)
heading      VARCHAR(200)
tags         JSON                   # 标签数组
content      VARCHAR                # 原文（用于 rerank 输入）
vector_dense BINARY_VECTOR(1024)   # BGE-M3 稠密
vector_sparse SPARSE_FLOAT_VECTOR   # BGE-M3 稀疏（BM25 视角）
```

索引：稠密用 `HNSW`（M=16, efConstruction=200），稀疏用 `SPARSE_INVERTED_INDEX`。

## 五、API 设计

### 5.1 通用约定

- Base：`/api`（生产经 Nginx 转发到 FastAPI）。
- 鉴权头：
  - 公开接口：`X-Session-Id: <anon_id>`（用于会话历史与限流）。
  - 后台接口：`Cookie: admin_token=...`（httpOnly）。
- 响应 envelope：

  ```json
  { "code": 0, "data": {...}, "message": "ok" }
  ```

  错误 `code != 0`，`message` 可直接展示。
- 分页：`?page=1&size=20`，响应 `{ items, total, page, size }`。
- 时间：ISO 8601 with timezone。

#### 5.3.1 看板聚合 `/api/admin/stats/overview`

前端看板（`web/app/admin/page.tsx`）顶部需要 4 个卡片指标 + 1 个「最近问答」列表。为避免前端发 N 次请求，本接口**单次聚合**返回所有看板所需数据。

**响应 schema（`StatsOverview`）**

```jsonc
{
  "code": 0,
  "data": {
    "summary": {
      "postCount": 6,           // 已发布文章数（status='published'）
      "questionCount": 128,     // 历史问答次数（chat_messages.role='user'）
      "knowledgeDocCount": 24,  // 知识库文档数（knowledge_docs 全部 status）
      "monthlyViews": 1532      // 本月访问数（聚合 chat_sessions + post_views）
    },
    "trend": {
      "posts": [3, 0, 1, 2, 0, 0, 0],     // 近 7 天每日新发文章数
      "questions": [12, 8, 15, 22, 9, 18, 14]  // 近 7 天每日问答次数
    },
    "topArticles": [            // 热门文章 Top 5（按引用次数降序）
      { "slug": "...", "title": "...", "channelName": "...", "citationCount": 18 }
    ],
    "topQuestions": [            // 热门问题 Top 5（按出现次数降序）
      { "content": "深圳人才补贴怎么申请？", "count": 7 }
    ],
    "recentQuestions": [        // 最近 10 条问答会话（看板「最近问答」区块）
      {
        "sessionId": "uuid",
        "title": "深圳人才补贴怎么申请？",
        "personaName": "政策顾问",
        "scopeType": "channel",
        "scopeRef": "policy",
        "firstQuestion": "深圳人才补贴怎么申请？",   // 会话首问（chat_messages 第一条 role='user'）
        "lastAnswerSnippet": "本科 15000 元/年...",  // AI 最近回答前 80 字
        "messageCount": 4,
        "createdAt": "2026-07-22T10:30:00Z"
      }
    ]
  },
  "message": "ok"
}
```

**实现要点**
- 一次 DB 会话并行 4 个聚合查询（`asyncio.gather`），目标 P95 < 200ms。
- `monthlyViews` 在 P0 阶段无独立访问日志表，**先用 `chat_sessions` 本月新增数 + `posts` 本月发布数估算**；P3 接入独立 `post_views` 表后替换实现，前端字段不变。
- `trend` 用 `generate_series` 按 `created_at::date` 聚合，确保空日期补 0。
- `topArticles` 通过 `chat_messages.citations::jsonb` 解引用 `article_id` 后 group + count。
- `recentQuestions` 复用已有 `chat_sessions` 表，关联 `personas.name` 取角色名；子查询取每会话首问 + 末答。
- Redis 缓存 60s（key: `stats:overview`），写文章/发问答时主动失效。

#### 语音流式通话 `/ws/voice`

连接参数：

```
wss://host/ws/voice?session=<session_id>&scope=...&persona=<persona_id>
```

| 首字节 | 类型 | 方向 | 内容 |
|--------|------|------|------|

| `0x02` | ASR 文本增量 | S→C | JSON `{ text }` |

{ type: 'ai_start' }

{ type: 'error', message }
{ type: 'fallback_text' }         // 降级文字提示

## 六、RAG 管道设计

### 6.1 范围路由

```
请求 scope
  │
  ├─ global       → partition=[global]
  ├─ channel:xxx  → partition=[channel:xxx]
  └─ article:xxx  → filter: article_slug == xxx
```

> 全站与频道互不重叠，channel partition 仅含该频道内容；article 范围用 filter 而非独立 partition，避免冗余索引。

### 6.2 检索流程

```
用户问题
   │
   ▼
┌─────────────┐
│ Query 改写  │   ── LLM 简短改写：去指代、补上下文
└──────┬──────┘
       │
       ▼
┌────────────────────────────┐
│ BGE-M3 稠密 + 稀疏混合检索  │   ── top_k_dense=20, top_k_sparse=20
└──────┬─────────────────────┘
       │ 合并去重
       ▼
┌──────────────────────┐
│ bge-reranker-v2 精排  │   ── top_n=5
└──────┬───────────────┘
       │
       ▼
┌──────────────────┐
│ 上下文组装        │   ── 拼接 rerank 后片段 + 文章元信息
└──────┬───────────┘
       │
       ▼
┌────────────────────────────┐
│ LLM 流式生成 + 引用对齐    │   ── PydanticAI Agent stream，标注 [1][2] → citations
└──────┬─────────────────────┘
       │
       ▼
   token 流 → WS
       │
       ▼
   生成完 → 计算 followups → done
```

### 6.3 引用溯源

- LLM prompt 要求在引用片段处标注 `[n]`。
- 生成完成后，后端解析 `[n]` 与 rerank top_n 的 chunk 对齐，组装 `citations`。
- `citation.snippet` 取 chunk content 的前 200 字符。
- 前端"查看原文"跳 `/posts/:slug#heading`。

### 6.4 主动澄清（clarify）

触发条件：
- 检索 top1 rerank 分数低于阈值（如 0.3）；
- 或问题包含歧义词且无明确频道上下文。

返回 `{ type: 'clarify', content: '你是想问 A 还是 B？', options: [...] }`，前端渲染为选项卡，点选项自动 `user_message`。

### 6.5 推荐追问（followups）

LLM 生成完成后追加一次轻量调用，基于本次问答产出 2-3 个候选问题。失败不影响主流程。

### 6.6 拒答兜底

- 命中敏感词 → 返回固定话术。
- 检索无结果（rerank 全部低于阈值）→ "我还没写过相关内容，你可以问我 XX 方向的问题。"
- LLM 超时/限流 → 降级为"根据文章摘要…"简短回答。

### 6.7 Agent 架构与 Skill 设计（v2.2）

**编排框架：PydanticAI**

与 FastAPI 同团队、Pydantic v2 原生、流式 stream、`@tool` 装饰器封装 Skill、`asyncio.Task.cancel()` 实现可中断。**不引入 graph interrupt 机制**——本项目的"中断"都是任务取消或对话轮次切换，不需要 checkpoint 恢复。

**Agent 形态：单 Agent + 线性管道 + clarify 分支**

不是"自主循环决策"，而是"管道式编排 + clarify 分支"：

```
user_message
   │
   ▼
query_rewrite (LLM 短调用，去指代/补上下文)
   │
   ▼
retrieve (@tool: BGE-M3 稠密+稀疏 → Milvus)
   │
   ▼
rerank (@tool: bge-reranker-v2-m3)
   │
   ▼
score_check ── 低分/歧义 ──► clarify 帧（特殊响应，本轮结束）
   │ 高分通过
   ▼
agent.run_stream (PydanticAI Agent, 流式 token, 标注 [1][2])
   │
   ▼
align_citations (@tool: [n] → citations)
   │
   ▼
generate_followups (@tool, 轻量 LLM 调用)
   │
   ▼
done 帧
```

**Skill 清单（封装为 PydanticAI `@tool`）**

| Skill 名 | 职责 | 输入 | 输出 |
|---------|------|------|------|
| `retrieve` | 稠密+稀疏混合检索 Milvus | scope, query, top_k | chunks[] |
| `rerank` | bge-reranker-v2-m3 精排 | chunks[] | top_n chunks |
| `align_citations` | 解析 [n] 标注对齐 chunk | text, chunks | citations[] |
| `generate_followups` | 基于 QA 生成 2-3 个追问 | question, answer | questions[] |
| `policy_check` | 敏感词/合规兜底 | text | safe/flagged |

**中断与取消（不使用 graph interrupt）**

| 场景 | 机制 |
|------|------|
| clarify 反问 | 作为 `clarify` 响应帧返回，**本轮结束**；用户选项作为新一轮 `user_message` 重入 Agent（天然对话循环） |
| 用户 `stop` | WS 收到 stop 帧 → `asyncio.Task.cancel()` 取消正在进行的 `agent.run_stream` |
| 语音打断 | VAD 检测开口 → 取消 TTS 推送 + 取消 LLM 生成（同上） |

**模块边界**

```
backend/app/rag/
  ├── provider.py        # LLM provider 抽象（通义/DeepSeek），封装为 PydanticAI model
  ├── embedding.py       # TEI 客户端（BGE-M3 稠密+稀疏）
  ├── reranker.py        # TEI 客户端（bge-reranker-v2-m3）
  ├── milvus_client.py   # pymilvus async 封装
  ├── chunker.py         # 标题层级 + token 滑窗分块
  ├── parser.py          # 多格式解析：MD(正则) / PDF(pypdf) / DOCX(python-docx) / TXT(直读)
  ├── tools.py           # @tool 封装：retrieve/rerank/align_citations/generate_followups/policy_check
  ├── agent.py           # PydanticAI Agent 定义（system_prompt + tools + 输出类型）
  ├── pipeline.py        # 编排：query_rewrite → retrieve → rerank → check → run_stream → citations → followups
  └── clarify.py         # 阈值判定 + 选项生成
```

**MCP 暂缓（P0+/P1）**

不引入 MCP。外部知识源（如政策开放 API）先用普通 `@tool` 直连；待出现"多外部源动态挂载"需求再抽 MCP server。

### 6.8 Agent v2：Query Understanding + FAQ-boosted + 问题库（v2.3）

> 变更(v2.3)：从"单 Agent + 线性管道"升级为"Query Understanding + FAQ-boosted 检索 + 问题库预计算"。新增 Q&A 生成管道（入库时 LLM 自动生成问题+答案，前端审核后写入 Milvus）。followups 从问题库取，替代关键词模板。

**动机**

v2.2 的管道存在三个问题：
1. 检索用原始 query embedding，未做关键词增强，召回率受限
2. 无 FAQ 机制——常见问题每次都要走完整 RAG（检索→rerank→LLM 生成），首字延迟高且答案不稳定
3. followups 基于关键词模板（`fallback.py`），质量低且与知识库无关

**新流程（2 次 LLM 调用）**

```
user_message
   │
   ▼
┌─ Round 0: Query Understanding（1次LLM结构化输出，小/快模型）────┐
│  输出 JSON: {keywords, intent, route, enhanced_query}            │
│  - keywords: 2-5 个核心关键词，用于增强 embedding 检索            │
│  - intent: 一句话描述用户意图                                     │
│  - route: faq_first | rag | channel | external_tool              │
│  - enhanced_query: 原始 query + keywords 拼接，用于检索            │
│  LLM 不可用时降级为原始 query + route=rag（不阻断流程）            │
└──────────────────────────────────────────────────────────────────┘
   │
   ▼
┌─ Round 1: 路由检索 ──────────────────────────────────────────────┐
│  route=faq_first → retrieve_faq()（chunk_type=qa）                │
│                   FAQ 无结果时回退到 retrieve_articles()           │
│  route=rag       → retrieve_faq() + retrieve_articles()           │
│  route=channel   → retrieve_articles()（频道 partition 范围）      │
│  route=external  → retrieve_articles() + 预留 @tool 外部调用      │
└──────────────────────────────────────────────────────────────────┘
   │
   ▼
rerank（FAQ + 文章片段一起精排）
   │
   ▼
score_check ── 低分/歧义 ──► clarify 帧（本轮结束）
   │ 高分通过
   ▼
FAQ 短路检查：top-1 是 Q&A 且 score >= 0.8 → 标记短路
   │
   ▼
build_context（FAQ answer + 文章片段合并，标注 [n]）
   │
   ▼
agent.run_stream（PydanticAI Agent, 人设模板, 流式 token, 标注 [1][2]）
   │
   ▼
align_citations（[n] → citations）
   │
   ▼
fetch_followups_from_faq（从问题库搜相关问题，排除已引用）
   │  无匹配时回退到关键词模板
   ▼
DONE
```

**FAQ 短路策略**

FAQ 高分命中（rerank score >= 0.8）时，Round 2 仍跑人设 LLM（保持人设语气一致性），但 context 主要是 FAQ answer，生成快且准。非短路时 FAQ + 文章片段合并作为上下文，FAQ 加权。

**数据模型变更**

Milvus collection `inkgrid_chunks` 新增字段：
- `chunk_type` VARCHAR(16)：`article_chunk`（文章片段）| `qa`（预生成 Q&A）
- `answer` VARCHAR(8192)：Q&A 答案（仅 qa 类型，article_chunk 为空）

Q&A 检索时 `content` 存问题文本（用于 embedding），`answer` 存答案。

PG 新表 `qa_pairs`：
- `id` UUID PK
- `article_id` FK → posts（CASCADE）
- `question` TEXT
- `answer` TEXT
- `status`：pending | approved | rejected
- `milvus_chunk_id` VARCHAR(64)：审核通过后写入 Milvus 的 chunk id

**Q&A 生成管道（Celery 异步）**

```
文章发布/更新
   │
   ▼
ingest.on_post_publish（入库）
   │
   ├─ ingest_article（解析→分块→embedding→写 Milvus）
   │
   └─ qa.generate_for_article（delay 异步触发）
        │
        ├─ 读取文章标题 + 正文
        ├─ LLM 生成 5 个 Q&A（JSON 结构化输出）
        └─ 写入 PG qa_pairs（status=pending）
           │
           ▼
        前端审核（admin/qa 页面）
           ├─ approve → reindex → 写入 Milvus（chunk_type=qa）
           ├─ reject  → 不入库
           └─ edit    → 修改后重新审核
```

文章删除时级联清理：PG qa_pairs + Milvus qa 向量。

**Skill / @tool 扩展点**

| 类型 | skill | 调用方式 | 阶段 |
|------|-------|---------|------|
| 内部 | `retrieve_faq` | 过程式（按 route） | 现在 |
| 内部 | `retrieve_articles` | 过程式 | 现在 |
| 内部 | `retrieve_channel` | 过程式 | 现在 |
| 外部 | `web_search` | PydanticAI `@tool` | 未来 |

Agent 预留 `tools` 参数注册点（`register_tool()`），当前为空（过程式路由），未来可注册 `@tool` 实现 LLM 自主 tool-calling（演进为 ReAct）。

**模块边界更新**

```

  ├── query_understanding.py  # Round 0: LLM 结构化输出（keywords + intent + route）
  ├── retriever.py            # FAQ-boosted 检索（retrieve_faq + retrieve_articles）

## 七、自动入库管道

### 7.1 触发时机

| 事件 | 动作 |
|------|------|
| 文章发布（draft → published） | 异步入库 |
| 文章更新（已发布） | 删旧 chunks + 重新入库 |
| 文章下架（published → archived） | 删除对应 chunks |
| 文章删除 | 删除对应 chunks + knowledge_docs |
| 手动上传文档（单文件或多文件批量） | 逐个入库（source_type=upload），源文件归档 MinIO |
| 政策条目录入 | 入库（source_type=policy，归属 channel） |

### 7.2 处理流程

```
入库事件（Celery task）
   │
   ▼
┌──────────────┐
│ 源文件归档    │  ── 上传文件(md/txt/pdf/docx) → MinIO
│ (仅 upload)  │     raw_uri = docs/{YYYYMM}/{uuid}{ext}
│              │     article 类型跳过此步（不存源文件）
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 文档解析      │  ── 按 source_format 分派：
│              │     md   → parse_markdown (正则提取 # 标题层级)
│              │     txt  → parse_text (直接返回纯文本，无标题层级)
│              │     pdf  → parse_pdf (pypdf 逐页提取文本)
│              │     docx → parse_docx (python-docx 按 heading 样式切层级)
│              │     article 类型固定走 parse_markdown
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 分块          │  ── 按标题层级 + 滑窗，目标 350 token/块，重叠 50
│              │     保留 heading 元信息写入 metadata
│              │     TXT/PDF 无标题层级时退化为纯滑窗分块
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 写入 PG       │  ── chunks 表 + knowledge_docs.parsed_text
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 批量 Embedding│  ── TEI 服务（BGE-M3 稠密+稀疏），batch=16，HTTP 调用
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 写入 Milvus   │  ── partition = channel:{slug}
│              │     knowledge_docs.status = indexed
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 同步 Meilisearch│  ── 用于前端即时搜索
└──────────────┘

### 7.2.1 上传接口与多文件处理

**文章页上传**（`POST /api/admin/posts/upload`）：
- 接收字段：`files: List[UploadFile]` + `channel_id: UUID`（Form）
- 仅接受 `.md` / `.markdown`，每文件走 6 道安全校验
- 不存源文件（符合现有约束），直接解析为 `posts.content_md` 创建草稿
- 返回批量结果：`{created: ArticleAdmin[], failed: [{filename, reason}]}`

**知识库上传**（`POST /api/admin/knowledge/upload`）：
- 接收字段：`files: List[UploadFile]` + `channel_id: UUID` + 可选 `title`（Form）
- 接受格式：`.md` / `.txt` / `.pdf` / `.docx`，按扩展名分派校验器与解析器
- **源文件归档 MinIO**：所有格式上传后先写 MinIO，`raw_uri` 存对象键
- 安全校验按格式差异化：
  - md/txt：UTF-8 解码、NUL 字节检测、5MB 上限
  - pdf/docx：二进制魔数校验（PDF: `%PDF`，DOCX: PK zip 头）、20MB 上限
- 同步入库（不派发 Celery，接口内完成解析+分块+embedding+Milvus 写入）
- 返回批量结果：`{created: KnowledgeDoc[], failed: [{filename, reason}]}`

**多文件错误隔离**：单个文件失败不阻断其他文件，失败原因逐个记录在 `failed` 列表。

### 7.2.2 文档解析依赖

多格式解析需在 `pyproject.toml` 的核心 dependencies 中新增：

| 依赖 | 用途 | 选择理由 |
|------|------|---------|
| `pypdf>=4.0` | PDF 文本提取 | 纯 Python 无系统依赖，轻量；不支持表格/图片提取（P0 不需要） |
| `python-docx>=1.1` | DOCX 文本与标题层级提取 | 按 heading 样式切层级，与 MD 的 `#` 标题层级一致 |

不引入 `pdfplumber`（重，P0 不需要表格）、`pymupdf`（AGPL 许可证）、`unstructured`（过重）。

**MinIO 通用文件操作**（`app/services/storage.py` 扩展）：
- `upload_document(content: bytes, content_type: str, filename: str) -> str` — 通用文件上传，对象键 `docs/{YYYYMM}/{uuid}{ext}`，返回对象键（非公开 URL，docs bucket 设私有读）
- `download_object(object_name: str) -> StreamingResponse` — 流式下载，用于 `/knowledge/docs/:id/download`
- `delete_object(object_name: str) -> None` — 删除对象，用于 knowledge_doc 删除时清理 MinIO

### 7.3 失败重试

- Embedding / Milvus 写入失败 → 任务重试 3 次，指数退避。
- 最终失败 → `knowledge_docs.status = failed`，后台可见，可手动 reindex。

### 7.4 全量重建

`POST /api/admin/knowledge/rebuild`：

- 加分布式锁 `lock:reindex:global`。
- 新建临时 collection，全量灌入，确认后 alias 切换。
- 期间问答走旧 collection，无停机。

## 九、语音通话编排

### 9.1 状态机

```
        ┌─────────┐ start_call  ┌────────────┐
        │  idle   │ ──────────► │ connecting │
        └─────────┘             └─────┬──────┘
            ▲                         │ WS 建立 + 麦克风授权
            │ end_call                 ▼
        ┌───┴──────┐ user_开口  ┌──────────┐ ai_开口  ┌──────────┐
        │  ended   │ ◄───────── │ listening │ ──────► │ speaking │
        └──────────┘            └──────────┘         └────┬─────┘
              ▲                                          │
              └─────────── user_开口打断 ────────────────┘
```

### 9.2 编排流程

```
音频帧 (0x01) 流入
   │
   ▼
┌─────────────────────┐
│ silero-vad 状态判定  │  ── 判断说话起止
└──────┬──────────────┘
       │ utterance_end (停顿 >600ms)
       ▼
┌─────────────────────┐
│ 流式 ASR             │  ── 边收边转写，增量发 0x02 给前端
└──────┬──────────────┘
       │ asr_final
       ▼
┌─────────────────────┐
│ RAG 检索 + LLM 流式  │  ── 复用文字问答的 rag 模块
└──────┬──────────────┘
       │ token 流
       ▼
┌─────────────────────┐
│ 流式 TTS             │  ── 边收 token 边合成，0x04 推给前端
└──────┬──────────────┘
       │
       ▼
   ai_end
```

### 9.3 打断机制

- VAD 检测到用户开口 → 发 `interrupt` 控制帧。
- 后端立即停止 TTS 推送（清空播放队列）。
- 如果 LLM 还在生成，可选：丢弃剩余 token 或转后台用于下次上下文。

### 9.4 回声消除

- 前端 `getUserMedia` 强制 `echoCancellation: true`。
- 后端 ASR 不再做回声消除（依赖浏览器）。
- 这是硬性要求，前端文档已强调。

### 9.5 降级

| 故障 | 降级 |
|------|------|
| 麦克风授权失败 | 发 `fallback_text`，引导回文字 |
| ASR 持续无输出 5s | 发 `status: listening` 重试提示 |
| LLM 超时 | 用 RAG 命中片段直接 TTS 朗读 |
| TTS 失败 | 文字字幕兜底 |
| WS 断连 | 前端重连 1 次，失败回退文字 |

### 9.6 限流

- 语音成本高，限流更严：单 anon_id 每日通话总时长 ≤ 30 分钟。
- 超限 → `error, code: VOICE_QUOTA_EXCEEDED`，引导回文字。

## 十、限流与安全

### 10.1 多层限流

| 层级 | 维度 | 限制 | 存储 |
|------|------|------|------|
| IP | 全局 | 60 req/min | Redis 滑动窗口 |
| IP | 问答 | 20 问/min | Redis 滑动窗口 |
| anon_id | 问答 | 50 问/天 | Redis 计数 |
| anon_id | 语音 | 30 min/天 | Redis 计数 |
| 后台 | 登录 | 5 次/min | Redis 计数 |

超限返回 `429`，WS 帧 `error, code: RATE_LIMITED`，含 `reset_at`。

### 10.2 验证码

- 触发条件：单 IP 1 分钟内问答 > 10 次，或被识别为爬虫特征。
- 实现：hCaptcha 或自研滑块；前端弹窗，验证后写 Redis 标记 1 小时免验。

### 10.3 合规兜底

- 输入侧：敏感词过滤（政治、暴力、色情等），命中直接拒答。
- 输出侧：LLM 输出再做一次敏感词检查，命中则替换为兜底话术。
- 政策频道：答案必须标注"以官方公告为准"。

### 10.4 鉴权

- 公开问答：仅靠 `X-Session-Id` 区分会话，不做身份校验。
- 博主后台：`argon2` 哈希密码，登录后 `Set-Cookie admin_token; HttpOnly; Secure; SameSite=Strict`。
- Token：JWT，过期 24h，刷新机制。
- 所有 `/api/admin/*` 走 middleware 校验。

### 10.5 审计

- 后台所有写操作记入 `audit_logs`。
- 公开问答异常（拒答、限流触发、敏感词命中）记入结构化日志。
- 日志保留 90 天。

### 12.1 关键指标

| 指标 | 目标 |
|------|------|
| 文章列表 API P95 | < 100ms |
| 文章详情 API P95 | < 150ms（含 content_html 缓存） |

| 语音端到端延迟 | < 1.8s（ASR+LLM+TTS） |
| 入库延迟（小文章） | < 5s |

### 12.2 缓存策略

- 文章详情：Redis 缓存 `cache:post:{slug}`，TTL 1h，发布时主动失效。
- 频道列表：缓存 5min。
- 热门文章：ZSET 每日更新。
- LLM 回答：对高频常见问题缓存（同一问题 + 同一 scope 24h 内复用）。

### 12.3 日志与追踪

- 全部请求经 middleware 注入 `request_id`，贯穿日志 + WS 帧。
- 结构化日志（structlog），输出 JSON。
- WS 连接日志：连接建立、断开、异常帧、限流触发。
