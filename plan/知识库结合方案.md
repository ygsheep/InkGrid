# 知识库结合方案：私有知识库 ↔ 公开博客的闭环

> 版本：v1.1 ｜ 日期：2026-07-24
> 状态：K1/K2/K3/K5 已完成，K4 AI 自动化与 Obsidian 互通待实施
> 依据：`plan/笔记.md`（Obsidian + Claude Code 本地知识管理）× `plan/产品与技术方案.md`（博客 + AI 问答）
> 变更范围：在现有 v2.1 博客体系基础上，引入"笔者私有知识库"，博客文章成为知识库的公开投影。

## 一、结合本质

将 `笔记.md` 的"Obsidian 本地存储 + Claude Code CLI 编译"升级为 **Web 多笔者知识库 + 后端 AI 自动化**；`产品与技术方案` 从"博客文章即知识库（唯一来源）"升级为 **"知识库 = 私有全集；博客 = 知识库的公开投影"**。

一句话定位：**每个笔者登录后拥有一个带 7 层目录的私有知识库；其中任意笔记可"发布"为公开博客文章；访客问答基于知识库（含未发布笔记）作答，并反哺知识缺口。**

### 与原两方案的关系

| 维度 | 笔记.md（原） | 产品方案（原） | 结合后 |
|------|---------------|----------------|--------|
| 存储 | Obsidian 本地 .md | 博客文章 = 知识库 | Web 私有知识库 + 公开博客投影 |
| AI | Claude Code CLI | 文章发布自动入库 | 后端 Celery 自动化（消化/MOC/分析/合成） |
| 公开口 | 无 | 博客文章 | 笔记发布为文章 |
| 闭环 | 收集→编译→存储 | 发布→问答 | 收集→AI编译→知识库→发布→问答→反馈回流Inbox |

### 闭环示意

```
笔者私有区 · 个人知识库（登录后可见）
  Markdown编辑器 ──写入──▶ 7层目录知识库 ◀──整理/链接── AI自动化引擎
                            │
                  发布为文章 │ ▲ 反馈回流（知识缺口→Inbox）
                            ▼ │
公开访客区 · 博客与问答（无需登录）
  博客文章 ──供给──▶ 频道问答 ──切换──▶ 语音对话
```

## 三、数据模型

### 3.0 存储架构（重要）

**Markdown 源码存 PostgreSQL，不存 MinIO；7 层目录是数据库虚拟的，不是真实文件系统。**

| 数据 | 存储位置 | 说明 |
|------|----------|------|
| 笔记正文 `content_md` | PostgreSQL `notes.content_md` (Text) | 单一数据源，搜索/RAG/双链解析直接走 DB |
| 图片/附件 | MinIO（复用现有 `upload_image` 链路） | bucket `inkgrid`，object 前缀 `images/` |
| 7 层目录结构 | DB 虚拟（`category` enum + `folder_path` 字符串树） | 不是真实文件夹，导出时按字段拼路径 |
| 向量 | Milvus | 复用现有 RAG 入库 |

**为什么 md 不存 MinIO**：
- 现有架构已是纯 DB 存储（`posts.content_md`），无需迁移
- DB 全文检索、双链 `[[标题]]` 解析、RAG 入库都直接查 DB，性能最好
- MinIO 双写会带来一致性问题
- 导出 zip 是 DB → 文件系统的"视图"，秒级生成，体验无损

**为什么目录是虚拟的**：
- 真实文件系统目录会限制查询能力（无法按 tags/status 聚合）
- `category` + `folder_path` 字段组合可完整表达 7 层结构，且支持任意维度查询
- Obsidian 导出/导入时按字段拼路径即可，对用户透明

### 3.1 notes 表（扩展/取代 articles）

在现有 `articles` 基础上扩展为通用笔记表：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int / pk | 主键 |
| owner_id | int | 笔者归属（本期单笔者，预留多租户） |
| category | enum | 7 层枚举：`inbox`/`daily`/`reading`/`knowledge`/`projects`/`templates`/`assets` |
| folder_path | str | 子目录路径，如 `knowledge/大模型`；仅 knowledge/projects 使用树结构 |
| title | str | 标题 |
| slug | str | URL 友好标识（发布时自动派生，复用现有逻辑） |
| content_md | text | Markdown 源码 |
| content_html | text | 渲染缓存（可选） |
| tags | jsonb | 属性标签数组，如 `["#大模型/微调","#type/教程"]` |
| status | enum | `draft` / `private` / `published`；`published` 才进博客 |
| published_at | ts | 发布时间 |
| channel_id | int | 发布绑定的频道（可空 = 全站） |
| is_moc | bool | 是否为主题地图节点 |
| source_url | str | 阅读笔记/采集来源 |
| excerpt | str | 自动派生（复用现有） |
| reading_time | int | 自动派生（复用现有） |
| toc | jsonb | 自动派生（复用现有） |
| created_at / updated_at | ts | 时间戳 |

**要点**：
- `00_Inbox/01_Daily/02_Reading/05_Templates/06_Assets` 用 `category` 枚举即可（扁平）。
- `03_Knowledge/04_Projects` 需要子目录树 → 用 `folder_path` 字符串树。
- **"发布" = 把 `status` 从 `private` 改为 `published` 并绑定 `channel_id`**，同一份内容，不复制。博客列表只查 `status=published`。
- 复用现有 Milvus 入库链路，新增 `chunk_type` 区分 `published` / `private`；公开问答 `scope` 仅检索 `published`。

## 五、前端：高效 Markdown 编辑体验

### 5.1 编辑器选型：Bytemd（维持原方案）

| 候选 | 内核 | 模式 | 适配度 | 结论 |
|------|------|------|--------|------|
| **Bytemd** | CodeMirror | 分屏预览 | 90% | **选用** |
| Vditor | 自研 | 三模式（含 Typora 式 IR） | 70% | 体积大、SSR 重 |
| Milkdown / Tiptap | ProseMirror | WYSIWYG 块级 | 55% | 偏离 .md 源码理念 |

**选 Bytemd 理由**：
1. **CodeMirror 内核** → 双链 `[[ ]]`、标签 `#` 的输入提示可复刻 Obsidian 体验。
2. **插件化 + SSR 友好** → 与 Next.js SSG/SSR 兼容，与展示端 MDX 渲染一致（同一套 remark/rehype 生态）。
3. **Markdown 源码派** → 忠于笔记.md 的纯文本可移植理念，未来导出 .md 无损。
4. 唯一短板"非即时渲染"可接受——技术博主习惯分屏，且可后续加 IR 模式插件。

### 5.2 参考产品底层

| 产品 | 编辑器内核 | 模式 | 启示 |
|------|-----------|------|------|
| Obsidian | CodeMirror 6 | 即时渲染 + 源码 | 双链/标签/图谱标杆，本地 .md 可移植 |
| Logseq | CodeMirror | 大纲即时渲染 | 验证双链可行 |
| Notion | 自研块编辑器（ProseMirror 系） | WYSIWYG 块级 | 体验现代，但非源码派 |
| 语雀 / 飞书文档 | 自研块编辑器 | WYSIWYG | 重投入 |
| 掘金 / 博客平台 | Bytemd 或 CodeMirror 系 | 分屏 | 与本场景最接近 |

### 5.3 关键能力实现

| 能力 | 实现方案 |
|------|----------|
| 源码 + 预览分屏 | Bytemd 原生 |
| **双向链接 `[[笔记标题]]`** | ① remark 插件：解析 `[[xxx]]` 为内部链接节点，渲染为站内跳转；② CodeMirror 插件：输入 `[[` 时弹笔记搜索浮层（查 notes 表），选中插入；③ 保存时后端解析 `[[标题]]` 写入 `note_links` 表，支撑反链面板 |
| **标签 `#一级/二级`** | CodeMirror 插件：输入 `#` 弹标签补全（查现有 tags）；remark 插件着色；侧栏标签云点击筛选 |
| 模板套用 | 新建笔记时从 `note_templates` 选模板，注入 content_md；05_Templates 即模板库 |
| 图片/附件 | 复用现有 MinIO 粘贴/拖拽上传链路（P0 已完成） |
| 草稿自动保存 | 复用现有防抖保存（P0 已完成） |
| 反链面板 | 编辑器右侧组件，读 `note_links` 显示引用当前笔记的笔记列表 |
| 快速捕获 | 顶部"闪念"入口 → 弹窗输入 → 存入 00_Inbox，标 `#status/待处理` |
| 大纲/折叠 | Bytemd 插件 |

双链是这套编辑器的灵魂，核心工作量在 **CodeMirror 输入提示插件 + remark 解析插件 + note_links 维护**三处，其余多为复用。

## 十、Obsidian 互通：导出 / 导入 zip

> ⛔ **状态：未实施**。下文为设计方案，尚未编码。后端 `kb_exchange.py` 服务、`/admin/kb/export.zip` 与 `/admin/kb/import.zip` 端点、前端导入/导出入口均待开发。

**目标**：Web 知识库 ↔ 本地 Obsidian 双向无缝切换。笔者在 Web 上整理，一键导出 zip，本地 Obsidian 直接打开；本地改完再上传 zip 导回。

### 10.1 设计原则

- **md 存 DB，zip 是视图**：导出 = DB 查询 → 按字段拼目录 → 打包；导入 = 解压 → 按目录解析 → upsert 入库
- **保留 Obsidian 原生语义**：`[[双链]]`、`#标签`、`![](图片)` 在导出/导入时保持原样，Obsidian 直接识别
- **图片路径双向转换**：DB 里存 MinIO URL，导出时换成相对路径 `06_Assets/...`，导入时换回 MinIO URL

### 10.2 导出（`GET /admin/kb/export.zip`）

生成的 zip 结构：

```
笔记库.zip
├── 00_Inbox/降低论文AIGC率.md
├── 01_Daily/2026-07-24.md
├── 02_Reading/认知觉醒.md
├── 03_Knowledge/大模型/高效微调.md         ← folder_path → 真实目录
├── 03_Knowledge/大模型/MOC - 大模型.md     ← is_moc 笔记
├── 04_Projects/InkGrid/设计方案.md
├── 05_Templates/日报模板.md
├── 06_Assets/202607/xxx.png                ← 从 MinIO 下载放入
└── CLAUDE.md                                ← 核心宪法（系统配置导出）
```

转换规则：

| DB 字段 | zip 中的体现 |
|---------|-------------|
| `category=inbox` | 目录 `00_Inbox/` |
| `category=knowledge` + `folder_path=knowledge/大模型` | 目录 `03_Knowledge/大模型/` |
| `title` | 文件名 `高效微调.md` |
| `content_md` | 文件内容（原始 markdown，含 `[[双链]]` `#标签`） |
| `content_md` 里的 MinIO 图片 URL `http://localhost:9000/inkgrid/images/202607/xxx.png` | 相对路径 `06_Assets/202607/xxx.png`，图片文件本身从 MinIO 下载放入 `06_Assets/` |
| `tags` | 文件内顶部 YAML front matter 或保留行内 `#标签` |
| `is_moc=true` | 文件名 `MOC - 主题.md` |
| 系统配置（AI 角色定义） | `CLAUDE.md` |

### 10.3 导入（`POST /admin/kb/import.zip`）

流程：
1. 解压 zip
2. 遍历目录树，按 7 层前缀映射 `category`：
   - `00_Inbox/*` → `category=inbox, folder_path=null`
   - `03_Knowledge/大模型/xxx.md` → `category=knowledge, folder_path=knowledge/大模型`
   - `06_Assets/202607/xxx.png` → 上传 MinIO，记录 URL 映射
3. 每篇 .md：
   - 文件名（去 `.md`）→ `title`
   - 正文里的相对图片路径 `06_Assets/...` → 替换为 MinIO URL
   - 解析 `[[双链]]` → 写 `note_links`
   - 解析 `#标签` → 写 `tags`
   - upsert 入 `notes` 表（按 `title` + `category` + `folder_path` 去重，已存在则更新 `content_md`）
4. `CLAUDE.md` → 写入系统配置

冲突策略（按用户选择）：
- **跳过**：已存在的笔记不动
- **覆盖**：用 zip 内容覆盖 DB
- **重命名**：冲突笔记加后缀 `-2`

### 10.4 限制与边界

- 单次 zip ≤ 100MB，笔记数 ≤ 2000（防滥用）
- 仅笔者本人可导出/导入自己的知识库
- 导入是异步任务（Celery），完成后通知笔者
- 导出可按 `category` 或 `status` 筛选（如只导出 published，或只导出 knowledge）
